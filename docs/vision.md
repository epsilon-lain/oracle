# Oracle Project Vision

Oracle is a lightweight AI-assisted tabletop room for running, organizing, and preserving role-playing stories.

It is not just a solo chatbot, and it is not just a generic story generator. Oracle is designed as a tabletop room where players can bring their own worlds, import or write scenario packs, create characters, roll dice, play with an AI game master, preserve campaign memory, and eventually export the adventure as readable story material.

The final feeling should be:

> You played a tabletop adventure, and Oracle helped turn it into a story you can read afterward.

## Core Vision

Oracle is built around one idea:

> Bring your own world, play the adventure, preserve the memory, and turn the campaign into a story.

Oracle should support both solo play and, in the long term, shared rooms for friends. A player or room host should be able to provide their own story materials, choose or write a scenario pack, configure the tone of the campaign, and let the AI game master help run the session.

The project should be lightweight enough to feel personal, but structured enough to keep a long-running campaign coherent. The room should hold the table, the characters, the dice, the current scene, the unresolved threads, and the archive of what changed.

Oracle itself should not ship copyrighted tabletop books, official campaigns, or proprietary settings. Instead, it should provide a flexible format for users to import their own homebrew adventures, open-source scenarios, legally owned notes, custom worlds, or AI-generated campaign drafts.

## Scenario Packs

A scenario pack is a structured story package that tells the AI game master how to run an adventure.

A scenario pack may include:

- setting premise
- tone and genre
- opening scene
- major locations
- important NPCs
- factions
- quests
- secrets
- enemies
- items
- rules assumptions
- dice expectations
- safety boundaries
- possible endings

The pack should be both human-editable and machine-readable. Users should be able to write one manually, generate one with AI, or import compatible open materials.

Scenario packs should give the AI game master enough context to understand what kind of story it is running without pretending to own the source material. Oracle's job is not to own every story. Oracle's job is to understand and run the stories users bring into the room.

## User-Owned Worlds

Oracle should be built around user-owned material.

Users may bring:

- homebrew worlds
- open-source scenarios
- self-written notes
- legally owned campaign material
- AI-generated campaign drafts
- original characters
- custom rules assumptions
- session notes from previous games

The system should help structure these materials into something an AI game master can actually use. A loose text dump may be useful as reference, but a structured scenario pack should be better for play.

Oracle should make it clear that users are responsible for what they import and that the product itself is a room and memory engine, not a library of copyrighted tabletop content.

## AI Game Master

The AI game master has two roles.

The first role is the front-stage DM. This is the part players talk to directly. It describes scenes, plays NPCs, asks for rolls, reacts to choices, manages tension, and keeps the story moving.

The front-stage DM should:

- describe scenes clearly
- keep the chosen tone consistent
- play NPCs with memory
- ask for rolls when uncertainty matters
- respect player agency
- react to choices
- avoid solving the story for the player
- keep the adventure moving

The second role is the backstage archivist. This part quietly organizes the campaign while the game is running. It tracks what happened, what changed, what the players chose, what NPCs know, what secrets remain hidden, and which plot threads are unresolved.

The backstage archivist should:

- summarize sessions
- track important decisions
- remember character details
- update NPC state
- update location state
- maintain unresolved threads
- preserve world changes
- prepare useful context for later sessions

Players should feel like they are talking to a game master, while Oracle maintains a living story archive behind the screen.

## Campaign Memory

Oracle should remember more than raw chat messages.

It should organize the adventure into useful memory layers:

- character memory
- world memory
- NPC memory
- location memory
- quest memory
- unresolved plot threads
- important choices
- session summaries
- tone and style notes
- future hooks

This memory should help the AI game master stay consistent across long adventures.

Campaign memory should not just be a transcript. A transcript records what was said. Oracle should also understand what matters: who changed, what promises were made, what clues were found, what dangers remain, and what the story is quietly becoming.

## Dice and Rules

Oracle should support dice as part of the tabletop feeling.

The system does not need to fully automate every tabletop ruleset. Instead, it should support lightweight rules assumptions, clear checks, visible rolls, and enough structure for uncertainty to feel fair.

Rules should be configurable through scenario packs, rule packs, or room settings. Oracle should avoid hard-coding copyrighted systems or shipping proprietary rules content.

## Solo and Multiplayer Direction

Oracle should work first as a solo tabletop room.

The solo room should let a player create or import a character, start a scenario, play through scenes with an AI game master, and return later to continue.

In the long term, Oracle should be multiplayer-friendly. Shared rooms should allow friends to play in the same campaign space, with Oracle helping the group remember what happened and organize the story afterward.

Multiplayer should not come before the solo loop is strong. The room should first prove that it can remember, continue, and export a satisfying adventure for one player.

## Story Export

After playing, users should be able to export the adventure.

Possible export formats include:

- raw session log
- cleaned campaign log
- chapter-style prose
- character journal
- case file
- world timeline
- summary for the next session

Export should make the campaign feel valuable after the session ends. The story should not be trapped inside chat history. Oracle should help transform play into readable material that can be saved, shared, edited, or continued.

## Product Feeling

Oracle should feel like:

- a tabletop room
- a living campaign archive
- a quiet game master behind the screen
- a place to bring original worlds
- a story engine with memory

Oracle should not feel like:

- a generic chatbot wrapper
- a random story prompt toy
- a plain CRUD app
- a copyrighted campaign library
- a rules encyclopedia without atmosphere

The interface should support focus, atmosphere, and continuity. The product should make the user feel that their campaign exists as an object with memory, not as a disposable chat thread.

## North Star

Oracle is a lightweight AI-assisted tabletop room where users can bring their own worlds, play with an AI game master, preserve campaign memory, and export the adventure as a story.
